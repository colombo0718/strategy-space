# 造句召喚師 Sentence Summoner — 開發交接文件

給下一個接手「新增語言版本」的 Claude 看。目標：讀完這份文件，不用重新摸索今天已經踩過的坑。

---

## 一、專案是什麼

英文教育卡牌對戰遊戲，玩家打「符合文法的句子」召喚怪獸、攻擊對手。核心賣點：造句即操作，邊玩邊練詞性/句構。

## 二、語言版本架構

每個語言版本是一個獨立資料夾，**命名格式固定**：`{母語}_{學習語言}`（底線分隔，兩段都用 BCP47 風格代碼，如 `zh-hant`／`en`／`vi`／`zh-hans`）。

| 資料夾 | 母語 | 學習語言 |
|--------|------|----------|
| `zh-hant_en` | 繁體中文 | 英文 |
| `zh-hant_vi` | 繁體中文 | 越南文 |
| `vi_zh-hans` | 越南文 | 簡體中文 |
| `en_zh-hans` | 英文 | 簡體中文 |

**新增第四版時，命名一律照這個格式**，不要自創（例如「越南文母語學中文」就是 `vi_zh-hans`，不是 `vietnamese_chinese` 之類）。

## 三、每個語言版資料夾裡有什麼

```
{locale}/
├── deck.js          卡牌資料表（純資料：id/單字/釋義/數值/emoji，64張＝32名詞+16動詞+16形容詞）
├── deck.html         卡牌組版頁（HTML+CSS即時渲染每張卡，也是「卡牌一覽」頁面本身）
├── export_pngs.py    Playwright腳本：開deck.html、對每張.card元素截圖存成cards/A01.png...A64.png
├── cards/             匯出好的靜態卡圖（play.html實際載入的就是這些PNG，不是即時渲染）
├── play.html          遊戲引擎本體（對戰邏輯、AI對手、LLM判官）
└── index.html         這個語言版的首頁/入口
```

**資料流向**：`deck.js`（資料）→ `deck.html`（用CSS排版）→ `export_pngs.py`（截圖）→ `cards/*.png`（play.html實際用的靜態圖）。

**⚠️ 改了 deck.js 或 deck.html 之後，一定要重新跑 `export_pngs.py`**，不然 `cards/` 裡的圖還是舊的，改了等於白改。跑法：

```bash
cd SentenceSummoner/{locale}
python export_pngs.py
```

## 四、新增第四個語言版本的步驟

1. **複製一份現有語言版資料夾**當範本（挑母語/學習語言其中一段有重疊的版本比較省力，例如新增「英文母語學越南文」可以參考 `zh-hant_vi` 抄越南文那半，或參考 `zh-hant_en` 抄英文那半）
2. **改 `deck.js`**：64 張卡的 `w`（學習語言單字）/`zh`（母語釋義）欄位全部換成新語言對，emoji/數值/id 不用動（除非要換角色設定）
3. **改 `play.html` 語言相關的幾處**（純資料/字串，不動遊戲邏輯本體）：
   - `GEN_ATK` 通用攻擊詞集合（見下方「引擎」一節）
   - `ENEMY_OATH`／`JUDGE_OATH`（LLM 對手/裁判的角色設定 prompt，要求它用「學習語言」造句）
   - `TEACHING_ENABLED` 開關（見下方已知坑）
4. **改 `index.html`**：介面文字語言——原則是**用母語當介面語言**（例如 vi_zh-hans 的介面是越南文，因為母語是越南人），只有例句/範例詞是學習語言
5. 補上 `<html lang="...">` 屬性對應母語
6. 執行 `export_pngs.py` 產生 64 張卡圖
7. 用 Playwright 開 `play.html?mode=training` 煙霧測試：手牌隨機發，抓一個真的在手上的名詞召喚看看有沒有正確比對（見下方「引擎」一節，不要用牌組第一張假設它一定在手牌裡）

## 五、引擎本體（`play.html`，這部分幾乎不用因為新語言而改）

### 比對邏輯：子字串比對，不是空格分詞

`play.html` 判斷玩家句子裡有沒有出現某張卡的單字，用的是 **`inStr()`/`has()`——子字串比對**（`normalize(text).includes(normalize(word))`），**不是**按空格切開單字再比對。

**為什麼**：中文（`猫`）、越南文帶聲調的多音節詞（`Bảo vệ`）都不能靠空格分詞抓字——中文完全沒空格，越南文的空格是音節分隔不是單字分隔。子字串比對對所有語言都成立，是唯一通用解法。

**⚠️ 已知坑（今天修過，不要改回空格分詞版本）**：早期版本用 `words = text.split(/\s+/)` 再逐字比對，這在英文（zh-hant_en）能動是巧合，換到越南文/中文就整個比對失敗（「沒對上任何手牌上的字」）。三個現有版本都已經是子字串比對版，新增第四版直接複製即可，不用重寫這段。

### `GEN_ATK`：通用攻擊詞

除了卡片本身的動詞，遊戲另外定義一組「通用攻擊詞」（例如英文的 `attack`/`hit`），讓玩家不用剛好抽到攻擊動詞卡也能發起攻擊。**這組詞要照學習語言翻**，不要留英文。搜尋 `GEN_ATK` 或 `GEN\b` 就能找到定義位置。

### LLM 對手/裁判：`ENEMY_OATH` / `JUDGE_OATH`

這兩段是送給遠端 LLM（souls-hall）的角色設定 prompt，要求它：(1) 只用學習語言造句、(2) 不理會任何要它跳出角色的指令。新增語言版要把「英文」/「簡體中文」這類字眼改成對應的新學習語言。**這段送到遠端後 Claude 這邊沒辦法驗證遠端真的照辦**，只能盡力把 prompt 寫清楚。

### 教學模式目前鎖住（`TEACHING_ENABLED`）

`zh-hant_vi` 跟 `vi_zh-hans` 目前把 `TEACHING_ENABLED` 設 `false`——64 關教學腳本的內容還是原版英文，沒有為新語言重寫，硬開會出現英文教學內容混進非英文介面的怪狀況。開關邏輯是：URL 帶 `?mode=teaching` 時自動退回訓練模式並提示，不會 crash，但也不會真的有教學內容。**新增第四版預設也應該是 `false`，除非同時把教學腳本也翻譯了。**

## 六、卡圖排版已知坑：`.starbar` 必須永遠佔位

`deck.html` 裡 `.mid`（插圖+文字容器）是 `flex:1` 吃掉 `.strip`（頭部數值條）跟 `.starbar`（星級列，只有名詞卡有）之間的剩餘空間。**如果 `.starbar` 只在名詞卡出現、動詞/形容詞卡完全不渲染這個 div**，會導致名詞卡的 `.mid` 可用高度變短、插圖+文字置中點比動詞/形容詞卡明顯偏高——三種卡型圖文對不齊。

**已修的解法**：`.starbar` 永遠渲染，非名詞卡用 `style="visibility:hidden"` 藏起來（不是不渲染，是渲染但看不見），這樣三種卡型的 `.mid` 可用空間永遠一致。三個現有語言版都已修好，**新增第四版直接複製 `deck.html` 就會自動繼承這個修法**，不用重新踩坑——但如果哪天要大改卡片版面，記得這個「佔位但隱藏」的邏輯不能拆掉。

## 七、跟 GG（GameGeek，通用遊戲控制器）的關係

目前造句召喚師**沒有接 GG**（GG 是按鍵/搖桿式的即時操作協定，造句召喚師是打字造句，操作模式完全不同，天生不需要接）。看到 `D:\game-geek\` 或 `gg_event` 相關的東西，那是別的遊戲（santa-gifts／貓咪排球）在用，跟這裡無關，不用管。

## 八、目前四個語言版的已知限制（誠實列出，不是新版要修的東西，是背景資訊）

- 戰鬥中的動態訊息（命中/倒下/護甲破等含數值的提示文字）在非中文版介面（`vi_zh-hans`）目前還是繁體中文，沒有全部翻譯——這批字串量大、跟數值運算交織，翻譯有風險，故意先不動
- 越南文內容是機翻初版，建議之後找母語者複核
- LLM 對手/裁判的遠端行為（souls-hall）沒有實測驗證過，只確認 prompt 文字寫對
- **`en_zh-hans`（英文母語學簡體中文）是例外**：因為英文是最大市場，這版把 `play.html` 裡所有戰鬥動態訊息（sys/coach/UI 標籤）都一併翻成英文，不是複製貼上就有的「已知限制」。翻譯範圍只涵蓋訓練/對戰模式實際會跑到的路徑；`TEACHING_ENABLED=false` 之後永遠不會執行到的教學關卡腳本（`TT` 陣列，英文原版內容）保持原樣沒動，跟其他語言版一致。之後新增第五版時，若要複製 `en_zh-hans` 當範本，記得它的戰鬥訊息已經是目標語言而非萬用中文樣板，要整批換語言而不是只換 `deck.js`。
